# CLAUDE.md

## 統一コーディングスタイルフォーマット

- **インデント**: すべて2スペースを使用
- **クラス・メソッド定義の{**: 必ず改行してから配置
- **if, for, while等の制御文の{**: 改行せずに同じ行に配置
- **改行ポリシー**: 行数を減らすことを優先し、1行が長くなる場合でも引数指定ごとに改行するなどは行わない
- **ドキュメントコメント**: 全てのクラス・メソッドに以下の `/** */` 形式のドキュメントコメントを必須とする（引数等のドキュメントは不要、概要のみ記載）

    ```
    /**
     * 概要をここに記載する
     */
    ```

- **コメント**: 処理の実装においては初学者でも理解できるよう、できる限り冗長なくらいコメントをつけるようにしてください（処理の塊ごと・分岐ごとに「何をしているか」「なぜそうするか」を日本語で記載する）
- **絵文字の使用禁止**: ソースコード上で絵文字のようなシステム依存の文字は使用しないでください
- **作業進捗番号の埋め込み禁止**: Phase~などの作業の進捗番号は絶対に記載しないでください

## ワークフロー管理

### 1. 計画モードをデフォルトに

- 非自明なタスク（3ステップ以上またはアーキテクチャ判断が伴う場合）は必ず計画モードで開始する
- 作業が想定外の方向に進んだら、即座に停止して再計画する（強引に進めない）
- 計画モードは「作る前」だけでなく「検証ステップ」でも活用する
- 曖昧さを減らすために詳細な仕様を最初に書く
- 使用のあいまいな部分を自身で補完はせず、必ずユーザーに質問ツールを用いて確定する

### 2. サブエージェント戦略

- メインのコンテキストウィンドウをクリーンに保つため、サブエージェントを積極活用する
- 調査・探索・並列分析はサブエージェントに委譲する
- 複雑な問題には、サブエージェントを通じてより多くの計算資源を投入する
- 1サブエージェント = 1タスク の原則で集中実行させる

### 3. 自己改善ループ

- ユーザーから修正を受けた場合は必ず `tasks/lessons.md` にパターンを記録する
- 同じミスを防ぐためのルールを自分向けに書く
- ミス発生率が下がるまでこのlessonsを徹底的に改善し続ける
- セッション開始時に、該当プロジェクトに関連するlessonsを確認する

### 4. 完了前の検証

- 動作を証明せずにタスク完了とマークしない
- 関連する場合はmainブランチとの変更差分を確認する
- 「スタッフエンジニアがこれを承認するか？」と自問する
- テストを実行し、ログを確認し、正確さを実証してから完了報告する

### 5. 優雅さの要求（バランス重視）

- 非自明な変更に対しては、一度立ち止まって「より優雅な方法はないか？」と問う
- 修正がハック的に感じたら、「今知っていることを全て踏まえた上で、優雅なソリューションを実装せよ」と自問する
- 単純で明確な修正にはこのプロセスをスキップする（過剰設計しない）
- 提案する前に自分の作業を自ら批評する

### 6. 自律的バグ修正

- バグレポートを受け取ったら、手取り足取り聞かずにそのまま修正する
- ログ・エラー・失敗テストを自分で確認して解決する
- ユーザーのコンテキストスイッチングをゼロにする
- CIテストが失敗していたら、指示を待たずに自分で修正する

### 7. テスト駆動開発 (TDD) 必須

- **すべてのプロダクションコードはテストファースト**で書く。Red(失敗) → Green(成功) → Refactor のサイクルを守る。
- 新規モジュールを切る時は、対応するテストファイルを同時に作成すること。テストの無い `commit` は原則禁止。
- バグ修正時は「先に失敗を再現する回帰テストを書き、その後に直す」順序を守る。
- レビュー観点: 該当変更を取り消した時にレッドになるテストが存在するか。
- 単体テストで賄えない領域(DB 接続, HTTP 経路, マイグレーション)は **DB 依存テスト**としてマークし、後述の docker-compose 前提で走らせる。
- スキップしてよいのは「外部 DB / 外部サービスが手元に存在しない時のみ」。本筋ロジックの単体テストを skip するのは禁止。
